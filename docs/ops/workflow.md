# 작업 흐름

## 형상 관리: GitHub

- `main`은 항상 실행 가능한 상태를 목표로 한다.
- 기능 작업은 `codex/<linear-id>-short-name` 형태의 브랜치에서 진행한다.
- 큰 기능보다 작은 플레이 가능한 변경을 선호한다.
- Unity 바이너리 에셋은 Git LFS로 관리한다.

## 위키: Notion

Notion은 사람이 읽는 프로젝트 허브로 사용한다.

- 프로젝트 목표
- 게임 디자인 결정
- 회의/아이디어 메모
- 출시 체크리스트
- 플레이테스트 노트

## 이슈 관리: Linear

Linear는 실제 작업 큐로 사용한다.

- 마일스톤은 개발 단계로 관리한다.
- 이슈는 하루나 이틀 안에 끝낼 수 있는 크기로 쪼갠다.
- 이슈 설명에는 완료 조건을 넣는다.
- 구현 중 생긴 설계 결정은 Notion 또는 `docs/`에 남긴다.

## 권장 이슈 형식

```markdown
## 목표

## 범위

## 완료 조건

## 메모
```

## 첫 개발 순서

1. Unity 프로젝트 생성
2. 첫 씬과 카메라 구성
3. 적 경로 이동
4. 웨이브 스폰
5. 랜덤 타워 소환
6. 자동 공격
7. 승리/패배 조건
8. Android 빌드 검증
